Pesquisa técnica OAuth (sem foco em API key)

Rodízio de contas OpenAI Codex no OpenClaw

Data da pesquisa: 07/03/2026. Objetivo: usar 3 contas (OAuth ChatGPT/Codex) e trocar automaticamente quando uma bater limite.

Sem agente extra
Funciona no agente padrão.
Sem API key
Fluxo 100% focado em OAuth.
Operação contínua
Failover automático + fallback.

Resposta curta

Sim, é viável no OpenClaw.

  • Não precisa criar agente separado. Você pode usar o agente padrão.
  • O OpenClaw suporta múltiplos perfis OAuth por provider em auth-profiles.json.
  • Quando um perfil falha por limite/rate/billing, ele entra em cooldown/disabled e o runtime gira para outro perfil.
  • Se todos os perfis do provider falharem, ele cai para agents.defaults.model.fallbacks.

O que validar antes

  • Use somente contas suas e autorizadas, em conformidade com os termos do OpenAI/ChatGPT.
  • Evite logar a mesma conta simultaneamente em muitos clientes OAuth, pois refresh token pode invalidar sessão antiga.
  • Não compartilhe auth.json / tokens.

Fluxo visual do rodízio

1) Sessão inicia OpenClaw escolhe perfil com base em auth.order (ou round-robin).
→
2) Perfil fixado Session stickiness: mantém o mesmo perfil enquanto estiver saudável.
→
3) Erro de limite/auth Perfil entra em cooldown/disabled por janela de backoff.
→
4) Próximo perfil Runtime troca automaticamente para a próxima contaN.
→
5) Último fallback Sem perfil viável? usa agents.defaults.model.fallbacks.

Como configurar o rodízio (OAuth Codex) no agente padrão

1) Usar o agente padrão (sem criar agente novo)

Se você já usa o padrão, não precisa rodar comando aqui.

2) Fazer login OAuth de cada conta

Use o fluxo que você já utiliza no wizard: openclaw config → selecionar modelo OpenAI Codex OAuth → login na conta.

No fluxo nativo, o login costuma gravar em openai-codex:default. Por isso, após cada login você deve renomear para contaN para não sobrescrever a conta anterior.

openclaw config

3) Confirmar perfis gerados

openclaw channels list --json

Os IDs costumam seguir provider:email quando o e-mail está disponível.

4) Definir ordem de prioridade dos perfis

No ~/.openclaw/openclaw.json, configure auth.order:

{
  auth: {
    order: {
      "openai-codex": [
        "openai-codex:conta1@dominio.com",
        "openai-codex:conta2@dominio.com",
        "openai-codex:conta3@dominio.com"
      ]
    }
  },
  agents: {
    defaults: {
      model: {
        primary: "openai-codex/gpt-5.3-codex",
        fallbacks: ["qwen-portal/coder-model"]
      }
    }
  }
}

5) Comportamento em runtime (como funciona na prática)

  • A rotação é feita pelo runtime de auth/provider do OpenClaw, não por um agente LLM de rotação.
  • OpenClaw usa “session stickiness” (não alterna a cada request; fixa por sessão até reset/erro).
  • Erros de limite/rate entram em cooldown exponencial (1m, 5m, 25m, 1h cap).
  • Falha de billing pode marcar perfil como disabled com backoff maior (5h até 24h cap).
  • Quando um perfil falha, a próxima tentativa usa o próximo perfil em auth.order.

6) Forçar troca manual por sessão (quando precisar)

/model openai-codex/gpt-5.3-codex@openai-codex:conta1
/model openai-codex/gpt-5.3-codex@openai-codex:conta2

Sintaxe: /model <provider/model>@<profileId>. Para remover o pin e voltar à rotação automática, rode /model sem o sufixo @profile.

Como o rodízio funciona por baixo dos panos

1) Fonte de perfis

  • Os tokens OAuth ficam no arquivo ~/.openclaw/agents/main/agent/auth-profiles.json.
  • Cada conta precisa de um profileId único (ex.: openai-codex:conta1, conta2, conta3).

2) Ordem de tentativa

  • Se houver auth.order["openai-codex"], essa ordem é seguida primeiro.
  • Se não houver ordem explícita, o OpenClaw usa ordem automática (round-robin com regras internas).

3) Session stickiness

  • O perfil escolhido fica preso à sessão para estabilidade e cache.
  • Não há troca em toda requisição; a troca ocorre em erro, reset de sessão ou nova sessão.

4) Falha e failover

  • Rate limit/auth/billing podem colocar o perfil em cooldown/disabled.
  • Quando isso acontece, o runtime tenta o próximo perfil da ordem.
  • Se nenhum perfil do provider estiver utilizável, cai em fallbacks de modelo/provider.

Automação (script pronto)

Arquivo criado em:

~/Desktop/pesquisas/rodizio-contas-codex-openclaw/openclaw-codex-account-orchestrator.sh

Uso padrão (faz login OAuth e depois renomeia openai-codex:default para o próximo contaN):

~/Desktop/pesquisas/rodizio-contas-codex-openclaw/openclaw-codex-account-orchestrator.sh

Exemplo: se já existem conta1 e conta2, o novo default vira conta3.

Menu interativo (adicionar nova conta ou encerrar):

~/Desktop/pesquisas/rodizio-contas-codex-openclaw/openclaw-codex-account-orchestrator.sh --interactive

No menu interativo também existe a opção de trocar a conta da sessão via /model ...@profile por baixo dos panos.

Modo em lote (ex.: adicionar 3 contas):

~/Desktop/pesquisas/rodizio-contas-codex-openclaw/openclaw-codex-account-orchestrator.sh --count 3

O que o script faz internamente

  • Dispara o login OAuth nativo do OpenClaw para Codex.
  • Lê auth-profiles.json do agente alvo e detecta quantas contaN já existem.
  • Renomeia automaticamente openai-codex:default para o próximo ID livre (contaN).
  • Atualiza referências internas (profiles, usageStats, lastGood) para manter consistência.
  • Cria backup do arquivo antes de cada alteração.

Isso já resolve seu cenário?

Para “acabou limite da conta A, usar B e depois C”, sim.

  • Você pode aplicar tudo no agente default.
  • A rotação entre perfis OAuth é nativa do OpenClaw.
  • Você controla prioridade com auth.order.
  • Pode combinar com fallback de modelo/provider.

Alternativas sem API key (se OpenAI esgotar)

  • Qwen OAuth via plugin oficial do OpenClaw (qwen-portal-auth), com device-code flow.
  • Fallback multi-provider no agents.defaults.model.fallbacks para manter continuidade.
  • Conta secundária via ChatGPT switcher existe no produto ChatGPT (até 2 ativas por sessão), mas não é orquestrador automático do OpenClaw.

Limitações reais encontradas

  • O Codex CLI oficial ainda não mostra suporte nativo maduro para múltiplas contas por “perfil” no binário principal; há issue aberta pedindo isso.
  • No OpenClaw, esse gap é contornado com auth-profiles.json + rotação/failover por provider.
  • Se o problema for política/termo de uso da conta, rodízio técnico não corrige bloqueio de compliance.
